# 병목 문장 독해 실험

한국어 과학 지문의 **문법적 병목 강도**가 읽기 시간과 이해도에 미치는 영향을 측정하는 웹 실험 도구.

**실험 페이지:** https://shingidong.github.io/reading_language/

---

## 핵심 설계

참가자는 시작 화면에서 **읽을 난이도(상 / 중 / 하)를 직접 선택**한다.
선택한 수준에 따라 **두 지문(열팽창 · 삼투)을 모두 그 수준으로** 읽는다.

**진행 흐름 (지문별 순차):**

```
지문1 읽기(명제 5개) → 지문1 문항 3개  →  지문2 읽기(명제 5개) → 지문2 문항 3개  → 제출
```

- 같은 내용(명제)을 **문장 구조만 바꿔** 상/중/하 세 버전으로 만들어 둠
- 지문 = 명제 5개 → 두 지문이면 참가자 한 명이 **명제 10개**를 읽음
- 문항은 **지문당 3개**(회상·통합·추론 각 1개), 두 지문 합쳐 **총 6문항**
- 지문 제시 순서는 무작위 (순서 효과 상쇄)

### 변인

- **독립변인**: 문장의 **병목 점수** (1.0 ~ 10.0, 연속 변인) — 상/중/하는 라벨일 뿐
- **종속변인**:
  - **음절당 읽기 시간 (ms)** ← 글자 수 차이를 보정한 값. **비교는 반드시 이 값으로!**
  - 재읽기 횟수
  - 이해 문항 정답 여부 (문항마다 근거 명제가 지정되어 있음)
  - 확신도 (확신함 / 잘 모름 / 찍었음) → '이해 착각'(오답인데 확신) 탐지

---

## ⚠️ 참가자가 난이도를 직접 고르는 설계의 한계 (보고서에 꼭 쓰세요)

이 실험은 참가자가 상/중/하를 **스스로 선택**합니다. 이건 요청하신 방식이지만,
해석에 중요한 제약이 따릅니다.

> 국어를 잘하는 학생이 '상'을 고르고 못하는 학생이 '하'를 고르는 경향이 있으면,
> "상이 어려워서 정답률이 낮은 것"인지 "원래 실력이 낮은 학생이 하를 골라서 정답률이 높은 것"인지
> **구분할 수 없습니다** (자기선택 편향).

따라서 이 데이터로는 **"병목 문장이 이해를 떨어뜨린다"는 인과 주장은 할 수 없고**,
**"난이도 상을 선택한 사람들은 이러이러한 지표를 보였다"** 정도의 기술(記述)이 가능합니다.

**보완 방법 (여력이 되면):**
- 국어 등급을 함께 기록하므로, **같은 등급 안에서** 상/중/하를 비교하면 편향이 줄어듭니다
  (`participants` 시트에서 국어등급으로 필터 후 비교).
- 참가자에게 상/중/하를 **모두** 풀게 하거나, 무작위로 배정하면 인과 추론이 가능해집니다.
  (원하면 이 방식으로 다시 만들 수 있습니다.)

## 병목 점수는 어떻게 매기는가

각 문장의 문법 자질을 세어 가중합으로 계산한다. ([data.js](data.js) 참고)

```
병목 점수(원점수) = 2.0 × 안긴절 수
                 + 1.5 × 명사화 수
                 + 1.2 × 피동 표현 수
                 + 0.8 × 절 밀도(절 수 ÷ 문장 수)
```

이후 30개 버전(2지문 × 5명제 × 3수준) 전체를 **1.0~10.0으로 정규화**한다.

| 자질 | 왜 병목인가 |
|---|---|
| 안긴절 | 미완결 구조를 작업기억에 쌓아둬야 함 |
| 명사화 | 용언을 명사로 압축해 사건 구조를 숨김 (부피가 변함 → 부피 변화) |
| 피동 | 행위자를 숨겨 누가 무엇을 했는지 흐림 (조절된다, 서술된다) |
| 절 밀도 | 한 문장에 절을 몇 개나 욱여넣었는가 |

### 수준별 평균 병목 점수 (검증 완료)

| 난이도 | 하 | 중 | 상 |
|---|---|---|---|
| 평균 병목 점수 | **1.8** | **4.5** | **7.9** |

모든 명제(10개)에서 하 < 중 < 상 단조 증가를 확인했습니다.

```
reading_language/
├── index.html            # 4개 화면 (시작 / 읽기 / 문항 / 완료)
├── app.js                # 실험 로직 (난이도 선택, 시간 측정, 채점, 전송)

└── .nojekyll
```

## 데이터 수집 설정 (필수)

`config.js`의 `SHEETS_WEBHOOK_URL`이 **비어 있으면 응답이 저장되지 않습니다.**

1. 구글 드라이브에서 **새 스프레드시트 생성**
2. **확장 프로그램 → Apps Script**
3. [google_apps_script.js](google_apps_script.js) 코드 전체를 붙여넣기 (기존 코드 삭제)
4. **저장 → 배포 → 새 배포**
   - 유형: **웹 앱** / 실행 계정: **나** / 액세스 권한: **모든 사용자** ← 반드시!
5. 발급된 URL을 `config.js`에 붙여넣기

```js
const SHEETS_WEBHOOK_URL = "https://script.google.com/macros/s/여기/exec";
```

```bash
git add config.js && git commit -m "웹훅 설정" && git push
```

## 데이터 분석

### ★ `analysis` — 이 시트 하나로 분석이 끝납니다

**한 행 = 참가자 한 명이 명제 하나를 읽은 기록.** 30명이면 300행이 쌓입니다.

| 참가자ID | 선택난이도 | 지문 | 명제ID | **병목점수** | 음절수 | **음절당읽기(ms)** | **해당정답률(%)** |
|---|---|---|---|---|---|---|---|
| 3-5-KDS | 상 | 삼투 현상 | 4 | 10.0 | 48 | 20.8 | 0 |
| 3-5-KDS | 상 | 열팽창… | 1 | 4.4 | 46 | 21.7 | 100 |
| … | | | | | | | |

**바로 할 수 있는 것:**
1. **산점도** — x축 `병목점수`, y축 `음절당읽기시간` → 우상향하면 가설 지지
2. **산점도 2** — x축 `병목점수`, y축 `해당정답률` → 우하향하면 가설 지지

```
# 병목 점수와 읽기 시간의 상관
=CORREL(analysis!I:I, analysis!Q:Q)

# 병목 점수와 정답률의 상관
=CORREL(analysis!I:I, analysis!U:U)

# 회귀 기울기 (병목 1점당 늘어나는 음절당 ms)
=SLOPE(analysis!Q:Q, analysis!I:I)

# 난이도별 평균 음절당 읽기 시간
=AVERAGEIF(analysis!E:E, "상", analysis!Q:Q)
=AVERAGEIF(analysis!E:E, "하", analysis!Q:Q)
```

### 나머지 시트

- **quiz_responses** — 문항별 응답. 각 문항에 근거 명제의 병목 점수가 붙어 있음
- **participants** — 참가자별 요약 (국어등급으로 필터해 자기선택 편향 완화)
- **raw_json** — 원본 백업

## 결과 해석 가이드

| 관찰 | 의미 |
|---|---|
| 병목점수 ↑ → 음절당 읽기시간 ↑ | 문법 구조가 처리 부담을 늘린다 |
| 병목점수 ↑ → 정답률 ↓ | 처리 부담이 이해 실패로 이어진다 |
| 병목점수 ↑ → 읽기시간 ↑ 인데 정답률 유지 | 더 오래 읽어서 **보상**했다는 뜻 |
| 병목점수 ↑ → '이해 착각'(오답+확신) ↑ | 병목 문장은 **틀렸는데도 맞았다고 믿게** 만든다 |

(위 관찰은 자기선택 편향을 감안해 조심스럽게 해석할 것)
